# hyper-mcp Auth PRD

## Summary

`hyper-mcp` needs an account/auth layer for the hosted MCP service. The service should allow an administrator to register and unregister agent accounts. Each account authenticates to `/mcp` with JWTs signed by its private key. The server validates those JWTs against either an inline registered public JWK or a registered JWKS URL.

The account lifecycle endpoints themselves are protected by an admin trust root configured at deploy time: either an admin public JWK or an admin JWKS URL. No admin trust root means registration endpoints are disabled.

## Current State

Current service includes:

- `GET /` public landing page
- `GET /health` public health check
- `POST /mcp` MCP endpoint
- PGLite-backed ports:
  - data
  - cache
  - blob
  - queue
  - search

Current service does **not** yet include:

- account registry
- JWT validation
- JWKS lookup/caching
- tenant isolation
- scope enforcement
- account lifecycle audit logs

## Goals

1. Add admin-protected account registration.
2. Add admin-protected account unregistration.
3. Validate JWTs generated by registered account private keys.
4. Support account key material via:
   - inline public JWK
   - JWKS URL reference
5. Scope access by port with `{port}:{scope}` grants.
6. Prepare for tenant-isolated port access.

## Non-Goals for First Auth MVP

- OAuth authorization flows
- User/password login
- UI dashboard for accounts
- Multi-admin account management beyond the configured admin trust root
- Public self-registration
- Hard-delete account data by default

## Admin Trust Root

The account lifecycle endpoints are enabled only when the server has an admin trust root configured.

Support exactly one of:

```env
HYPER_MCP_ADMIN_PUBLIC_JWK='{"kty":"OKP","crv":"Ed25519","x":"...","kid":"admin-1"}'
```

or:

```env
HYPER_MCP_ADMIN_JWKS_URL=https://example.com/.well-known/jwks.json
```

Required metadata:

```env
HYPER_MCP_ADMIN_ISSUER=admin-agent
HYPER_MCP_ADMIN_AUDIENCE=hyper-mcp
```

Optional:

```env
HYPER_MCP_ADMIN_KID=admin-1
HYPER_MCP_JWKS_CACHE_SECONDS=300
```

### Admin Rules

- The admin key/JWKS is the bootstrap authority.
- The admin key/JWKS is **not** itself a normal account.
- Admin JWTs must include `accounts:admin` scope.
- Admin JWTs can call `/register` and `/unregister`.
- Registered accounts can call `/mcp` according to their scopes.

## Endpoints

## `POST /register`

Registers an account with either an inline public key or a JWKS URL.

### Auth

Requires:

```http
Authorization: Bearer <admin-signed-jwt>
```

Admin JWT must validate against configured admin public JWK or admin JWKS URL.

Admin JWT must include either:

```json
{ "scope": "accounts:admin" }
```

or:

```json
{ "scp": ["accounts:admin"] }
```

### Request

```json
{
  "accountId": "agent-tom",
  "name": "Tom's agent",
  "issuer": "agent-tom",
  "audience": "hyper-mcp",
  "publicJwk": {
    "kty": "OKP",
    "crv": "Ed25519",
    "x": "...",
    "kid": "agent-tom-1"
  },
  "jwksUrl": null,
  "ports": {
    "data:read": true,
    "data:write": true,
    "cache:read": true,
    "cache:write": true,
    "blob:read": true,
    "blob:write": true,
    "queue:read": true,
    "queue:write": true,
    "search:read": true,
    "search:write": true,
    "data:dangerous": false,
    "queue:dangerous": false,
    "search:dangerous": false
  }
}
```

### Rules

- Must provide exactly one of:
  - `publicJwk`
  - `jwksUrl`
- `ports` keys are normalized to scopes.
- Unknown ports/scopes are rejected.
- Default is deny.
- `false` values are omitted from granted scopes.
- Registration writes an audit log entry.

### Response

```json
{
  "ok": true,
  "accountId": "agent-tom",
  "issuer": "agent-tom",
  "audience": "hyper-mcp",
  "scopes": [
    "data:read",
    "data:write",
    "cache:read"
  ],
  "status": "active"
}
```

## `POST /unregister`

Disables an account. This should not hard-delete account data in the MVP.

### Auth

Requires:

```http
Authorization: Bearer <admin-signed-jwt>
```

Admin JWT must include `accounts:admin`.

### Request

```json
{
  "accountId": "agent-tom",
  "confirm": true
}
```

### Rules

- Requires `confirm: true`.
- Sets account status to `disabled`.
- Disabled account JWTs no longer work for `/mcp`.
- Unregistration writes an audit log entry.

### Response

```json
{
  "ok": true,
  "accountId": "agent-tom",
  "status": "disabled"
}
```

## Scope Model

```txt

```

Allowed ports:

```txt

```

Allowed scopes:

```txt
read
write
admin
dangerous
```

Examples:

```txt

cache:read
blob:write
queue:write
search:read
accounts:admin

```

## Proposed Database Tables

### `accounts`

- `account_id text primary key`
- `name text`
- `issuer text not null`
- `audience text not null`
- `status text not null` — `active | disabled`
- `scopes jsonb not null`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

### `account_keys`

- `account_id text references accounts(account_id)`
- `kid text`
- `public_jwk jsonb not null`
- `created_at timestamptz not null default now()`
- primary key: `(account_id, kid)`

### `account_jwks`

- `account_id text references accounts(account_id)`
- `jwks_url text not null`
- `cache_expires_at timestamptz`
- `cached_jwks jsonb`
- `created_at timestamptz not null default now()`

### `account_audit_log`

- `id text primary key`
- `actor text`
- `account_id text`
- `action text not null`
- `outcome text not null`
- `metadata jsonb`
- `created_at timestamptz not null default now()`

No private keys or raw JWTs should be stored.

## JWT Validation

For admin and account JWTs, validate:

- signature
- `iss`
- `aud`
- `exp`
- `nbf`
- `kid` when provided
- account status for normal accounts
- required scopes

Supported key sources:

- inline public JWK
- JWKS URL with TTL cache

## Implementation Plan

### Step 1 — Auth schema

Add account, key, JWKS, and audit tables to PGLite migrations.

Acceptance criteria:

- Schema migration is idempotent.
- Account/key records persist across restarts.
- Invalid account/key records are rejected.
- Existing port tests still pass.

### Step 2 — Admin trust root config

Read admin public JWK or admin JWKS URL from environment.

Acceptance criteria:

- Exactly one admin trust source is allowed.
- If neither is configured, `/register` and `/unregister` return `503 admin_not_configured`.
- If both are configured, startup fails with a clear config error.

### Step 3 — Admin JWT verification

Use `jose` to validate admin JWTs.

Acceptance criteria:

- Valid admin JWT with `accounts:admin` can call `/register`.
- Valid admin JWT without `accounts:admin` receives `403`.
- Expired JWT receives `401`.
- Wrong issuer receives `401`.
- Wrong audience receives `401`.
- Unknown `kid` receives `401`.
- JWKS URL is fetched and cached.

### Step 4 — Register endpoint

Implement `POST /register`.

Acceptance criteria:

- Admin can register an account with inline public JWK.
- Admin can register an account with JWKS URL.
- Request must include exactly one of `publicJwk` or `jwksUrl`.
- Ports object accepts only valid `{port}:{scope}` keys.
- `false` values are omitted from granted scopes.
- Registration writes an audit log.

### Step 5 — Unregister endpoint

Implement `POST /unregister`.

Acceptance criteria:

- Admin can disable account with `confirm: true`.
- Missing `confirm` returns `400`.
- Disabled account can no longer access `/mcp`.
- Unregistration writes an audit log.

### Step 6 — Protect `/mcp`

Validate normal account JWTs before MCP requests.

Acceptance criteria:

- `GET /` works unauthenticated.
- `GET /health` works unauthenticated.
- `POST /mcp` without token returns `401`.
- `POST /mcp` with valid account JWT reaches MCP transport.
- Disabled account gets `401` or `403`.

### Step 7 — Scope enforcement

Enforce port scopes for every MCP tool.

Acceptance criteria:

- Read-only token can read but not write.
- Port-specific token can access only that port.
- Missing required scope returns `403`.
- Dangerous operations require `{port}:dangerous` and existing `confirm`/dangerous gates.

### Step 8 — Tenant isolation

Scope all persisted port data by `account_id`.

Acceptance criteria:

- Account A cannot read Account B data.
- Same collection/key/blob/index/topic names can exist independently per account.
- Multi-tenant tests cover every port.

### Step 9 — Docs and examples

Update README with key generation, registration, JWT signing, and MCP examples.

Acceptance criteria:

- Developer can generate key pair.
- Developer can register an account.
- Developer can sign a JWT.
- Developer can call `/mcp` with the JWT.
- Render env setup is documented.

## Replacement semantics (Option A — full replace)

`/register` is the single account upsert endpoint. When it targets an existing
account, it **fully replaces** that account's auth material before installing
the new credential:

1. `accountCreate` upserts the account row (name, issuer, audience, scopes,
   status=`active`).
2. If the account already existed, `accountClearAuth` deletes every row in
   `account_keys` and `account_jwks` for that `accountId`, and an
   `auth_replace` audit entry is written.
3. The new credential is then stored — either `accountAddKey` (inline public
   JWK) or `accountAddJwksUrl` (JWKS URL), exactly one of which is required.

Consequences:

- Re-registering from an inline JWK to a JWKS URL removes the old inline key,
  so it can no longer authenticate.
- Re-registering from a JWKS URL to an inline JWK removes the JWKS URL, so it
  is no longer consulted.
- Single-mode re-registration (same mode, refreshed key) replaces the prior
  key/URL rather than accumulating them.
- A fresh registration (new accountId) clears nothing (no-op) and behaves as
  before.

This avoids stale-key / stale-JWKS ambiguity during auth-mode switches. An
`auth_replace` audit entry records each replacement with the new method.

Auth Port — users, sessions, passwords, one-time codes

Slug: auth-port  |  Branch: feat/auth-port  |  Inspired by: Convex Auth Library (session tokens, users, password + email/SMS OTP)  |  Status: planned

Summary

Add a new auth MCP data-plane port to hyper-mcp, scoped exactly like data/cache/blob/queue/search. An account calls /mcp with auth:* scopes and manages its own application's users, credentials, sessions, and one-time sign-in codes. Backed by PGLite and tenant-isolated by account_id. This makes "auth like data, blob, cache" literally true: an account can build auth for its app on top of hyper-mcp.

Project-specific definition

An auth port is a set of MCP tools under the auth scope prefix that manage end-user identity and sign-in state for the calling account's own application. It provides four primitives:

  • Users — identity records (email/username/phone + arbitrary attributes), unique within an account.
  • Passwords — salted scrypt hashes stored separately from user records; verify-only.
  • Sessions — opaque, random, url-safe session tokens with expiry; stored as SHA-256 hashes so a DB leak does not expose live sessions.
  • One-time codes — 6-digit email/SMS OTP codes with TTL, attempt cap, and single-use consumption.

What it is NOT:

  • NOT the server's own admin/account auth (control plane — already implemented).
  • NOT OAuth/social login flows (deferred — future auth_link_oauth could add this).
  • NOT JWT issuance by hyper-mcp; sessions are opaque tokens, not signed JWTs.
  • NOT email/SMS delivery; auth_create_code generates and returns the code — delivery is the account's responsibility.
  • NOT a login UI or dashboard.
  • NOT a change to the tool surface, scope model, or tenant model of existing ports.

Goals

  • Users: create, get, find by email/username, update attributes, delete (dangerous).
  • Passwords: set/verify with scrypt; hashes never returned.
  • Sessions: create (issue token), verify (resolve user), revoke, list for a user.
  • One-time codes: create (email/SMS), verify (consume), with TTL + attempt cap.
  • Tenant isolation by account_id, enforced and tested.
  • auth:read / auth:write / auth:dangerous scopes; readOnly blocks writes; dangerous blocks delete.
  • Conformance harness extended so future adapters must pass the auth suite.

Non-goals

  • OAuth/OIDC provider linking, social login, magic-link email delivery.
  • JWT session tokens (opaque tokens only for MVP).
  • Refresh tokens, session rotation, device tracking.
  • MFA enrollment flows, passkeys, WebAuthn.
  • Password reset email flows (the account orchestrates with auth_create_code).
  • Rate limiting / brute-force protection beyond the per-code attempt cap (deployments add their own).

Open questions / assumptions

#QuestionResolution
Q1Session token format?Opaque 32-byte random base64url string. Store SHA-256 hash; return plaintext only on creation. No JWT.
Q2Password hashing?Node built-in crypto.scrypt: 16-byte salt, 32-byte hash, N=16384, r=8, p=1, encoded as scrypt$<salt-b64>$<hash-b64>$<params>. No new dependency.
Q3Email/username uniqueness scope?Unique within an account (account_id + field), via partial unique indexes where the field is non-null. Different accounts may reuse the same email.
Q4Do auth_get_user/auth_find_users return credentials?No. Never return password hashes, session tokens, or code values. Only user identity + attributes + status.
Q5One code per target, or many?One active (non-consumed, non-expired) code per (account_id, target). auth_create_code replaces any prior active code for that target.
Q6Should auth_verify_session auto-expire?Yes: expired sessions are marked invalid (treated as not found). A reaper is not required for MVP; verify checks expires_at > now() and the revoked flag.
Q7Where do credentials live in the schema?A separate auth_credentials table keyed by (account_id, user_id, method), so user record reads can never leak hashes.

Security / privacy considerations

  • Passwords: scrypt-hashed with per-user salt; plaintext never stored or logged; auth_get_user excludes credentials.
  • Sessions: store only SHA-256(token); plaintext token returned once at creation; verify is constant-ish lookup by hash.
  • Codes: store SHA-256(code); TTL default 10m; max_attempts default 5; consumed on success; attempts increment on miss.
  • Errors are unauthenticated-impersonation-safe: auth_verify_password/auth_verify_session/auth_verify_code return { valid: false } rather than leaking which side failed.
  • Tenant isolation: every table keyed by account_id; all methods take accountId first arg like other ports.
  • readOnly blocks all auth:write tools; auth:dangerous (auth_delete_user) requires allowDangerous + confirm.
  • No secrets, hashes, tokens, or codes are logged in logger calls — only ids and outcomes.

Schema (PGLite)

CREATE TABLE IF NOT EXISTS auth_users (
  account_id text NOT NULL,
  user_id text NOT NULL,
  email text,
  username text,
  phone text,
  attributes jsonb NOT NULL DEFAULT '{}'::jsonb,
  status text NOT NULL DEFAULT 'active',
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (account_id, user_id)
);
CREATE UNIQUE INDEX IF NOT EXISTS auth_users_email_uq
  ON auth_users(account_id, email) WHERE email IS NOT NULL;
CREATE UNIQUE INDEX IF NOT EXISTS auth_users_username_uq
  ON auth_users(account_id, username) WHERE username IS NOT NULL;

CREATE TABLE IF NOT EXISTS auth_credentials (
  account_id text NOT NULL,
  user_id text NOT NULL,
  method text NOT NULL,            -- 'password'
  hash text NOT NULL,
  updated_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (account_id, user_id, method)
);

CREATE TABLE IF NOT EXISTS auth_sessions (
  account_id text NOT NULL,
  token_hash text NOT NULL,        -- sha256(token) hex
  user_id text NOT NULL,
  expires_at timestamptz NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  metadata jsonb,
  revoked boolean NOT NULL DEFAULT false,
  PRIMARY KEY (account_id, token_hash)
);
CREATE INDEX IF NOT EXISTS auth_sessions_user_idx
  ON auth_sessions(account_id, user_id);

CREATE TABLE IF NOT EXISTS auth_codes (
  account_id text NOT NULL,
  code_id text NOT NULL,
  channel text NOT NULL,           -- 'email' | 'sms'
  target text NOT NULL,            -- email address or phone
  code_hash text NOT NULL,         -- sha256(code) hex
  user_id text,                    -- optional binding
  expires_at timestamptz NOT NULL,
  attempts integer NOT NULL DEFAULT 0,
  max_attempts integer NOT NULL DEFAULT 5,
  consumed boolean NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (account_id, code_id)
);
CREATE INDEX IF NOT EXISTS auth_codes_target_idx
  ON auth_codes(account_id, target);

Step 3 — Passwords (scrypt)

Risk: medium Security-sensitive.

Work: Add scrypt hash/verify helpers (node:crypto). Implement authSetPassword/authVerifyPassword; register tools. authSetPassword upserts auth_credentials row with method 'password'. authVerifyPassword returns { valid: boolean }; missing user or missing credential → valid:false.

Acceptance criteria:

  • set then verify with correct password → valid:true; wrong password → valid:false.
  • verify for a user with no password set → valid:false.
  • auth_get_user never returns the hash; hash is stored only in auth_credentials.
  • Deleting a user removes their credential row.

Evidence: test/auth-port.test.ts password block.

Step 4 — Sessions

Risk: medium

Work: Implement authCreateSession/authVerifySession/authRevokeSession/authListSessions; register tools. Token = 32-byte base64url; store sha256 hex; default TTL via HYPER_MCP_AUTH_SESSION_TTL_SECONDS (default 86400). Verify hashes input, checks revoked=false and expires_at>now().

Acceptance criteria:

  • create returns { token, expiresAt }; verify with that token → { valid:true, userId }.
  • verify with wrong/tampered token → valid:false; revoked → valid:false; expired → valid:false.
  • auth_list_sessions returns active (non-revoked, non-expired) sessions for a user, without token hashes.
  • Tenant isolation: account A's token does not verify in account B.

Evidence: test/auth-port.test.ts sessions block.

Step 5 — One-time codes

Risk: medium

Work: Implement authCreateCode/authVerifyCode; register tools. 6-digit code; store sha256; default TTL 600s, max attempts 5. Create replaces any active code for the target. Verify looks up active code by (account_id, target), increments attempts on miss, marks consumed on hit. Returns { valid, userId? }.

Acceptance criteria:

  • create returns { code, expiresAt }; verify with correct code → valid:true and code consumed.
  • second verify of a consumed code → valid:false.
  • wrong code max_attempts times → further verify → valid:false (code unusable).
  • expired code → valid:false.
  • Tenant isolation: account A's code does not verify in account B.

Evidence: test/auth-port.test.ts codes block.

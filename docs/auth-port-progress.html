Residual risks / follow-ups

  • No rate limiting / brute-force protection beyond the per-code attempt cap; deployments must add throttling.
  • auth_verify_code looks up by target only (channel is metadata); create replaces all codes for a target regardless of channel. Acceptable for "one active code per (account,target)" but worth noting if multi-channel-per-target is ever needed.
  • auth_create_code does not validate that a bound userId exists; it is recorded as metadata only. Could add a check in a follow-up.
  • No OAuth/OIDC linking, social login, refresh tokens, session rotation, MFA, or passkeys — deferred by design.
  • Session tokens are opaque (not signed JWTs); no offline verification.

BaaS Adapter — functions, identity-at-boundary, user-scoped data

Slug: baas-adapter  |  Branch: feat/baas-adapter (off feat/auth-port)  |  Status: planned

North star

hyper-mcp is a swappable-substrate BaaS spoken over MCP: every capability lives behind a stable adapter contract so a user or agent builds an app against the contract and never commits to a backend vendor — prototype on lightweight local implementations, swap to production implementations without rewriting the app. Data/cache/blob/queue/search already follow this; control-plane auth and the auth port follow it; this PRD makes the last layer — BaaS itself — follow the same principle: end-user identity at the boundary, row-level user-scoped data, and running the app's own functions.

Result: an agent or user can build a web app on hyper-mcp (frontend on ZenBin) without writing or hosting a separate backend service. You host hyper-mcp; you register functions into it, Convex-style. When you outgrow the prototype substrate, you swap adapters, not the app.

Project-specific definition

BaaS adapter = three contracts plus a dynamic HTTP endpoint:

  • IdentityResolver — resolves a request credential to { accountId, userId | null }.
  • FunctionRuntime — executes an account's authored function with a scoped ctx and returns JSON.
  • User-scoped data — ctx.db that is auto-scoped to the verified user (the RLS surface).
  • Dynamic endpoint — POST /u/:accountId/:fn wires resolver → function → runtime, with public vs authed functions.

What it is NOT (this cycle): the production adapters are contract-only here — external-Postgres engine RLS, Daytona function sandbox, and OIDC JWT boundary auth are defined as the swappable prod impls but not implemented. This cycle ships the prototype impls end-to-end.

Research findings (gating)

  1. PGLite does NOT enforce RLS policies. Probed directly: ENABLE/FORCE ROW LEVEL SECURITY + CREATE POLICY … USING (owner = current_setting('app.user_id', true)) is accepted and stored (relrowsecurity=true, relforcerowsecurity=true), but queries still return all rows regardless of the setting. Conclusion: the prototype substrate cannot do engine RLS. Prototype enforces user-scoping in application code (the ctx.db capability wrapper stamps/filters user_id in SQL). The prod adapter (external Postgres) moves to engine-enforced RLS with SET LOCAL app.user_id. Same contract, different enforcement.
  2. Prototype function runtime = node:vm with a hard-restricted context. quickjs-emscripten v0.32 has a fiddly split API not worth the cycle; isolated-vm needs native build. node:vm is zero-dep and adequate for the prototype threat model (code authored by the account owner in dev). It is explicitly not a security boundary — loud warning in code, startup log, and docs; untrusted/multi-tenant code uses the Daytona FunctionRuntime (prod, contract-only here).

Function model

A function is authored as a JS expression evaluating to an async handler:

async (ctx) => {
  // ctx.user = { id, accountId } | null   (null for public functions)
  // ctx.db  = user-scoped data (get/find/create/update/delete/count)
  // ctx.auth = createUser/verifyPassword/createSession/getUser/findUsers
  // ctx.kv  = per-user JSON KV (set/get/delete) backed by the cache port
  const u = await ctx.auth.createUser({ email: ctx.body.email });
  await ctx.auth.setPassword(u.userId, ctx.body.password);   // note: see ctx shape
  const s = await ctx.auth.createSession(u.userId);
  return { userId: u.userId, token: s.token };
}

The runtime evals the source to obtain the handler and calls it with ctx. The HTTP request body is exposed as ctx.body. Functions return JSON-serializable values; non-serializable returns are rejected.

Goals

  • Define IdentityResolver, FunctionRuntime, and user-scoped data as adapter contracts with prototype impls.
  • Function store: register/get/list/delete functions (JS source + public flag), tenant-isolated by account.
  • POST /u/:accountId/:fn: resolve identity (skip if public), run function with scoped ctx, return JSON.
  • User-scoped data: ctx.db enforces user_id on every read/write; cross-user reads impossible from a function.
  • Function-management MCP tools so an agent can register functions.
  • End-to-end test: signup → login → authed profile/posts with RLS isolation between two users.

Non-goals (this cycle)

  • Prod adapters: external Postgres engine RLS, Daytona runtime, OIDC JWT boundary (contract-only).
  • Function versioning/rollbacks, canary, aliasing (store version but latest wins for MVP).
  • Network access from functions (ctx.fetch), cron, webhooks, file system, env vars.
  • Deno/TypeScript transpilation — functions are plain JS.
  • Per-function resource caps (CPU/memory/time) beyond a wall-clock timeout.
  • A function marketplace or cross-account sharing.

Schema (PGLite, additive)

CREATE TABLE IF NOT EXISTS app_functions (
  account_id text NOT NULL,
  name text NOT NULL,
  body text NOT NULL,            -- JS expression evaluating to async (ctx) => {...}
  public boolean NOT NULL DEFAULT false,
  version integer NOT NULL DEFAULT 1,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (account_id, name)
);

CREATE TABLE IF NOT EXISTS app_data (
  account_id text NOT NULL,
  user_id text NOT NULL,
  collection text NOT NULL,
  id text NOT NULL,
  document jsonb NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (account_id, user_id, collection, id)
);

app_data is intentionally separate from data_docs: account-level data (via /mcp data:* tools) and user-level data (via functions' ctx.db) are distinct. The wrapper enforces user_id in every query (prototype RLS). The existing data port is untouched.

Security / privacy

  • Prototype runtime is NOT a security boundary. node:vm restricted context is for code the account owner authored (dev/trusted). Startup logs and docs say so. Untrusted/multi-tenant requires the Daytona adapter. Do not run third-party functions on the prototype runtime in hosted mode.
  • ctx.db is the only data path from functions; it is impossible to address another user's user_id — the wrapper injects ctx.user.id and ignores any caller-supplied user id. Cross-user reads/writes are structurally prevented at the wrapper, not via a policy that can be misconfigured.
  • ctx.auth methods run with the route's accountId; a function cannot touch another account's users/sessions.
  • Public functions (e.g. login) run with ctx.user = null but still only affect the route's account.
  • Account JWTs are never exposed to functions or browsers; only opaque session tokens leave the server, and only via functions that choose to return them.
  • Function source is stored as text; secrets/keys are never injected into ctx. No process, require, env, Buffer, or network in the sandbox global.
  • Wall-clock timeout per function call; uncaught errors return a generic 500 to the caller (no stack trace leak by default in hosted mode).

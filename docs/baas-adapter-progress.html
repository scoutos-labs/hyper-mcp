Research findings (gating)

  • PGLite does not enforce RLS policies — ENABLE/FORCE ROW LEVEL SECURITY + policy accepted and stored but queries return all rows. Prototype enforces user-scoping in app code (ctx.db wrapper); prod external Postgres uses engine RLS.
  • Prototype runtime = node:vm restricted context — trusted dev code only; quickjs-emscripten v0.32 API too fiddly for this cycle; isolated-vm needs native build. Prod = Daytona (contract-only).

Security: ctx.db structurally prevents cross-user access; no Node globals in sandbox; account JWT never exposed (only opaque session tokens via functions); prototype-runtime caveat loud in code + startup log + docs; prod adapter path explicit.

Residual risks / follow-ups

  • Prototype runtime is not a security barrier. node:vm is escape-able; do not run untrusted functions on it in hosted mode. Daytona adapter (prod) is the follow-up.
  • RLS is app-level, not engine-level. PGLite can't enforce RLS; the ctx.db wrapper is the enforcement. A bug in the wrapper = a cross-user leak. The prod Postgres adapter moves to engine-enforced RLS policies.
  • Async functions are not killed on timeout — the endpoint returns 500 but the underlying promise may still resolve; no CPU/memory caps. Add real cancellation + resource caps (Daytona) for prod.
  • No ctx.fetch/network, filesystem, env, timers, or TypeScript transpilation for functions (deferred).
  • Function versioning: latest version wins; no aliasing/canary/rollback.
  • Account JWT app:* scope grants function management; the /u endpoint does not use account JWTs (user session tokens only) — so the broker-backend gap is closed for the function surface, but account-level /mcp data access still needs an account JWT.

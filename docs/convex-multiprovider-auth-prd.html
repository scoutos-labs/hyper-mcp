Multi-Provider Admin Trust Root — applying the Convex auth pattern to hyper-mcp

Slug: convex-multiprovider-auth  |  Source research: zenbin.org/p/convex-auth-widget-catalog  |  Status: planned

Summary

Convex auth (auth.config.ts) declares a list of trusted OIDC providers per deployment, each with its own JWKS URL and issuer, and verifies incoming JWTs by routing on the token's iss claim to the matching provider's JWKS. hyper-mcp today accepts exactly one admin trust root — one inline JWK or one JWKS URL, one issuer, one audience. This task brings hyper-mcp's admin trust layer to parity with the Convex pattern: a list of trusted admin providers with issuer-routed JWT verification, while preserving full backwards compatibility with the existing single-provider env vars.

Project-specific definition

"Multi-provider admin trust root" in hyper-mcp means: a deployment-time list of trusted admin identity providers. Each provider declares issuer, audience, and exactly one key source (inline publicJwk or remote jwksUrl), plus an optional kid. validateAdminJwt decodes the token's iss claim without verification, selects the matching provider, and then runs jose's jwtVerify against that provider's key set with the provider's own iss/aud. Scopes are still required server-side (accounts:admin); the server remains the authority.

What it is NOT:

  • Not a change to account authentication — accounts already support inline JWK and JWKS URL.
  • Not OAuth login flows, user/password login, or a dashboard UI.
  • Not a change to the MCP tool surface, scope mapping, port interfaces, or tenant model.
  • Not multi-tenancy of admins; admin JWTs always resolve to the synthetic admin account.
  • Not a replacement of jose; it reuses createLocalJWKSet/createRemoteJWKSet.

Goals

  • Accept a list of admin providers and verify admin JWTs by issuer-routed provider selection.
  • Backwards compatibility: existing single-HYPER_MCP_ADMIN_* deployments work unchanged.
  • Clear, fail-closed behaviour: ambiguous config or unknown issuer fails with explicit errors.
  • No auth bypass; no silent unauthenticated fallback.

Non-goals

  • Account-side auth changes (already JWKS-capable).
  • Changing endpoints, tool surface, scopes, or tenant isolation.
  • Admin RBAC / multi-admin tiers.
  • OAuth flows, refresh tokens, session management.

Open questions / assumptions

#QuestionResolution
Q1How to declare multiple providers via env?New JSON env var HYPER_MCP_ADMIN_PROVIDERS = a JSON array of provider objects. Mutually exclusive with the legacy single-provider vars; setting both is a startup error.
Q2Can two providers share an issuer?No. Issuer routing needs a 1:1 issuer→provider map. Duplicate issuers fail startup with a clear error.
Q3Keep config.admin field?Remove it; gate checks move to config.adminProviders.length. Removes a confusing alias and forces all call sites to use the list.
Q4Key-set caching across providers?Module-level Map<string, KeySet> keyed by a stable provider key string (iss|aud|publicJwkOrJwksUrl). Remote sets keep jose's built-in cooldown/cache.
Q5Issuer decoding before verification — safe?Yes: the unverified iss is only used to select which provider verifies; signature/aud/exp are still enforced by jwtVerify. This mirrors Convex's JWKS-by-issuer lookup.

Security / privacy considerations

  • No secrets, private keys, or token strings are logged. Errors reference issuer/kid only.
  • Fail closed: zero providers → 503 on /register//unregister, stdio refuses to start in hosted mode.
  • Ambiguous config (legacy + PROVIDERS) fails startup rather than silently picking one.
  • Issuer uniqueness enforced at config parse so routing cannot authenticate against the wrong provider.
  • Provider key sets are never leaked; only verification results surface.

Config format

# Legacy single provider (unchanged, still supported)
HYPER_MCP_ADMIN_PUBLIC_JWK='{"kty":"OKP",...,"kid":"admin-1"}'
HYPER_MCP_ADMIN_ISSUER=admin-agent
HYPER_MCP_ADMIN_AUDIENCE=hyper-mcp
HYPER_MCP_ADMIN_KID=admin-1

# OR new multi-provider list (mutually exclusive with the legacy vars)
HYPER_MCP_ADMIN_PROVIDERS='[
  {"issuer":"admin-agent","audience":"hyper-mcp","publicJwk":{"kty":"OKP",...,"kid":"admin-1"}},
  {"issuer":"clerk-admin","audience":"hyper-mcp","jwksUrl":"https://clerk.example.com/.well-known/jwks.json","kid":"clerk-1"}
]'

Tradeoff: a JSON env var is the simplest way to express a list in a 12-factor deployment without inventing indexed env names. It is slightly harder to hand-edit than scalar vars, but the legacy scalar path remains for single-provider setups, so the cost only falls on operators who actually need multiple providers.

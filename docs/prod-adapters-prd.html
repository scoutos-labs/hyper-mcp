Prod BaaS Adapters — Daytona, Postgres engine RLS, OIDC boundary

Slug: prod-adapters  |  Branch: feat/prod-adapters (off main)  |  Status: planned

Summary

Implement the production impls behind the BaaS adapter contracts shipped in PR #10, so the swappable substrate actually has a prod side — not just a prototype. Three adapters, each behind the contract already in src/baas/types.ts:

  • OIDC JWT IdentityResolver — boundary auth via Clerk/AuthKit/Auth0 JWTs verified against the provider's JWKS (reuses the jose machinery from the admin trust).
  • Daytona FunctionRuntime — real isolated execution of untrusted functions in Daytona sandboxes, with capability RPC back into hyper-mcp.
  • External Postgres engine RLS — engine-enforced row-level security for BaaS app_data via Postgres FORCE ROW LEVEL SECURITY policies + SET LOCAL app.user_id.

Adapter selection is config-driven; prototype impls remain the default. The full external-Postgres Ports adapter (all ports) is explicitly a follow-up — this cycle proves the prod RLS path on the BaaS app_data surface.

Definition

Each prod adapter is a drop-in impl of an existing contract (IdentityResolver, FunctionRuntime, AppDataPort) selectable by env config. The /u/:accountId/:fn endpoint and the function authoring DX are unchanged — only the backing implementation swaps.

What it is NOT: not a hosted SaaS (you still deploy hyper-mcp); not a new function DX; not the full external-Postgres Ports adapter (follow-up); not function versioning/cron/network (still deferred).

Research findings (gating)

  • No local Postgres in this env. External-Postgres RLS tests are gated on PG_TEST_URL; they skip when unset. (PGLite's lack of RLS enforcement was already established in the BaaS cycle.)
  • @daytona/sdk v0.192.0 is available. Daytona FunctionRuntime is implementable; real e2e is gated on Daytona env (DAYTONA_API_KEY etc.), orchestration is unit-tested with a mocked client.
  • jose already a dep — OIDC IdentityResolver is fully implementable + testable now with a local JWKS HTTP server (same pattern as test/admin-multiprovider.test.ts).

Adapter selection (config)

HYPER_MCP_BAAS_IDENTITY=opaque        # opaque (default) | oidc
HYPER_MCP_BAAS_RUNTIME=vm              # vm (default, trusted dev) | daytona
HYPER_MCP_APP_DATA_BACKEND=pglite      # pglite (default) | pg

# OIDC boundary providers (when identity=oidc): JSON list
HYPER_MCP_BAAS_OIDC_PROVIDERS='[
  {"issuer":"https://clerk.example.com","audience":"hyper-mcp","jwksUrl":"https://clerk.example.com/.well-known/jwks.json","accountId":"myapp"}
]'

# Daytona (when runtime=daytona)
DAYTONA_API_KEY=...            # plus DAYTONA_* env the SDK expects
HYPER_MCP_DAYTONA_TIMEOUT_MS=20000

# External Postgres app_data (when app_data_backend=pg)
PG_TEST_URL=postgres://user:pass@host:5432/db   # also used by gated tests
HYPER_MCP_APP_DATA_PG_URL=postgres://...

A provider's accountId binds an OIDC issuer to the hyper-mcp account whose functions end users may call. sub becomes the userId (the OIDC user id is the row key); if the user does not yet exist in auth_users, it is auto-provisioned on first call with attributes = { oidc_sub } so ctx.db has a stable user_id.

Plan

Step 1 — OIDC JWT IdentityResolver + config

Risk: medium

Work: src/baas/identity-oidc.ts: OidcIdentityResolver that decodes the unverified iss, selects the matching configured provider, jwtVerify against its JWKS (createRemoteJWKSet) with the provider's audience, then resolves accountId (provider-bound) + userId = sub (auto-provisioning the user via authCreateUser if missing, idempotent on attributes.oidc_sub). Add HYPER_MCP_BAAS_IDENTITY + HYPER_MCP_BAAS_OIDC_PROVIDERS config parsing. Resolver factory selects opaque vs oidc.

Acceptance: valid OIDC JWT resolves to {accountId, userId}; wrong issuer/audience/expired → null; user auto-provisioned once; second call reuses the same user. Tests use a real local JWKS server.

Step 2 — Internal capability RPC for sandboxed functions

Risk: medium Security-sensitive.

Work: Add internal endpoints POST /_internal/cap/:op (op ∈ db_get/db_find/db_create/db_update/db_delete/db_count/auth_*/kv_*) that accept a short-lived internal capability token (signed by hyper-mcp with a per-call secret) binding {accountId, userId, exp}. The Daytona runtime injects an RPC shim into the sandbox that calls these endpoints with the token. This keeps all data access server-side and RLS-scoped; the sandbox never sees a DB connection.

Acceptance: internal token verified; capability calls enforce the same user-scoping as ctx.db; token expires; rejected without/with-tampered token → 401. Unit-tested directly.

Step 3 — Daytona FunctionRuntime

Risk: medium

Work: src/baas/runtime-daytona.ts: DaytonaFunctionRuntime using @daytona/sdk. Per call: get/create a sandbox, inject the function source + an RPC shim (uses fetch to /_internal/cap/* with the internal token) + a JSON ctx payload ({user, body, capUrl, capToken}), execute, collect stdout/JSON result, enforce timeout, dispose/reuse sandbox. Unit-test orchestration with a mocked Daytona client; gate the real e2e on DAYTONA_API_KEY.

Acceptance: mocked end-to-end (shim → cap endpoint → app_data) returns the function's JSON result; timeout aborts; sandbox errors → 500. Real Daytona test skips when env unset.

Step 4 — External Postgres engine RLS for app_data

Risk: medium

Work: src/baas/appdata-pg.ts: PgAppDataPort implementing AppDataPort against external Postgres via pg. Schema migration creates app_data with FORCE ROW LEVEL SECURITY + policy USING (user_id = current_setting('app.user_id', true)) WITH CHECK (...). Every method runs in a transaction that does SELECT set_config('app.user_id', $userId, true) first, so the engine enforces scoping — the wrapper is no longer the enforcement. Factory selects pglite vs pg for app_data. Tests gated on PG_TEST_URL.

Acceptance: with a real Postgres (PG_TEST_URL set), cross-user reads/writes are blocked by the engine; a row with another user_id is invisible; a cross-user insert is rejected (WITH CHECK). Tests skip cleanly when PG_TEST_URL is unset.

Step 5 — Wire selection into the endpoint + startup

Risk: low

Work: app.ts builds the resolver/runtime/appdata from config (factories in src/baas/index.ts). The /u endpoint uses the configured resolver + runtime; ctx.db uses the configured appdata backend. Startup logs name the selected impls (and the vm-trusted warning only when vm is selected). Default = opaque/vm/pglite (unchanged behavior).

Acceptance: default config behaves identically to today; selecting oidc/daytona/pg routes through the prod impls; startup logs the choices.

Step 6 — Tests + docs + validation

Risk: low

Work: test/baas-oidc.test.ts (real JWKS server), test/baas-daytona.test.ts (mocked client + real gated), test/baas-appdata-pg.test.ts (gated on PG_TEST_URL), test/baas-internal-cap.test.ts (internal token + capability endpoints). README + port-contracts.md prod-adapter section. typecheck/build/test.

Acceptance: all ungated tests pass; gated tests skip cleanly; typecheck + build green; test count up; docs describe selection + prod impls + the full-Postgres-Ports follow-up.

Security / privacy

  • OIDC resolver: unverified iss only routes provider selection; signature/aud/exp enforced by jwtVerify; provider→account binding is config-trusted; sub is the user key. No JWT stored.
  • Internal capability tokens: short-lived, signed, bind {accountId, userId, exp}; the sandbox only gets this token + the cap URL, never DB credentials. Cap endpoints re-verify and enforce user-scoping server-side.
  • Daytona: functions run in an isolated sandbox with no Node host access; the only egress is the capability RPC to hyper-mcp. Real isolation — the security barrier the prototype lacks.
  • Postgres RLS: FORCE ROW LEVEL SECURITY + WITH CHECK; scoping is enforced by the engine per-transaction via SET LOCAL app.user_id, not by a wrapper that can be bypassed by a buggy query.
  • No secrets in logs; internal token never returned to the caller; process/require unavailable in the sandbox (Daytona runs only the injected script + shim).

Non-goals

  • Full external-Postgres Ports adapter (data/cache/blob/queue/search/auth on external Postgres) — follow-up; this cycle proves engine RLS on app_data.
  • Daytona sandbox pooling/keep-warm optimization (per-call create is acceptable for MVP prod; pooling is a follow-up).
  • Function network egress beyond the capability RPC (no arbitrary fetch to third parties).
  • OIDC refresh tokens / session rotation — boundary JWTs are short-lived from the provider.
